Configuring the sample application feed

This book documents the process of deploying a sample application called Octopub. Octopub implements a sample online library and is implemented as a number of microservices.

For convenience, the services that make up Octopub have been compiled, packaged, and exposed via a public Maven repository. In this chapter we'll configure an Octopus Maven feed which we'll use in later chapters to download and deploy Octopub to AWS.

Note that although Maven is typically associated with Java, we are not limited to deploying Java applications. Maven repositories can host arbitrary files, such as zip files. Maven feeds also have the advantage of being able to be accessed anonymously. This means the sample application can be distributed and consumed without sharing any credentials.

To create the feed in Octopus, apply the Terraform configuration in the terraform/feeds/octopus directory using the following commands: 

cd terraform/feeds/octopus
terraform init
terraform apply

You will be prompted for the URL of your Octopus instance, the Octopus API key, and the ID of the space to apply the configuration to. Terraform then displays a list of changes and prompts you to apply them. Once the configuration is applied you will see a new feed called Sales Maven Feed:

Breaking down the HCL

We start by defining a requirement on the Octopus terraform provider.

Note that the version of the provider is fixed. The Octopus provider is a zero-version release indicating breaking changes may be introduced at any time. By fixing the version of the provider, we can update it at our own pace, and deal with any potential breaking changes in an organized manner.

Had we referenced the latest version of the provider, our Terraform configuration may have randomly stopped working, which is not an ideal outcome: 

terraform {
    required_providers {
      octopusdeploy = { source = "OctopusDeployLabs/octopusdeploy", version = "0.10.3" }
    }
  }

The Octopus provider requires the Octopus server, API key, and space ID. Each of these values is consumed from a variable: 

provider "octopusdeploy" {
    address  = "${var.octopus_server}"
    api_key  = "${var.octopus_apikey}"
    space_id = "${var.octopus_space_id}"
  }

The Octopus server, API key, and space ID are defined as Terraform variables, allowing them to be passed in when applying the Terraform configuration: 

variable "octopus_server" {
    type        = string
    nullable    = false
    sensitive   = false
    description = "The URL of the Octopus server e.g. https://myinstance.octopus.app."
  }
  
  variable "octopus_apikey" {
    type        = string
    nullable    = false
    sensitive   = true
    description = "The API key used to access the Octopus server."
  }
  
  variable "octopus_space_id" {
    type        = string
    nullable    = false
    sensitive   = false
    description = "The ID of the Octopus space to populate."
  }

The terraform configuration block, provider initialization, and provider variables above are shared between all projects. We won't cover them again as they are identical ever time.

The final resource is the Octopus feed. We have used the Maven feed to access a public AWS S3 bucket, allowing Octopus to consume artifacts without having to define any credentials: 

resource "octopusdeploy_maven_feed" "sales_maven_feed" {
    download_attempts              = 10
    download_retry_backoff_seconds = 20
    feed_uri                       = "https://octopus-sales-public-maven-repo.s3.ap-southeast-2.amazonaws.com/snapshot"
    name                           = "Sales Maven Feed"
  }
